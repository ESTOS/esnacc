# Introduction

`node-server` is the Node.js backend for the `ts-microservice` sample. It is a TypeScript Express service that hosts generated ASN.1 ROSE stubs and exposes them to clients.

Features:

- Full TypeScript support (Node 24, `pnpm`)
- Express routes for REST invokes and WebSocket sessions
- Generated ASN.1 server stubs (`-RTS_SERVER`, `TSASN1Server`)
- Handler modules for the sample Settings and Event Manager interfaces

# Related packages

This server is used by two clients in the same sample:

| Package | Role |
| --- | --- |
| [`../browser-client`](../browser-client) | Interactive browser demo (`TSASN1BrowserClient`, Vite UI) |
| [`../node-client`](../node-client) | Automated integration tests (`TSASN1NodeClient`, `node:test`) |

Both clients share the same ASN.1 modules from `samples/interface/`. The server does not distinguish between them — only the transport and client stub differ.

## node-client (integration tests)

`../node-client` is a headless test runner, not a user-facing application. When you run `pnpm test` there:

1. A child `node-server` process is started with test-specific environment variables (port `13020`, log directory under `log/integration-tests`, JSON encoding).
2. `node:test` cases connect via REST and WebSocket and assert invoke results and inbound events.
3. The server process is stopped when tests finish.

This is how CI validates that a freshly built `esnacc` produces working TypeScript stubs and that the server wiring still matches the runtime. For local development of server handlers, you can still run `node-server` manually and use either `browser-client` or `node-client` against it.

# How to start

Prerequisites: Node 24, `pnpm` (via Corepack). From the `samples` folder run `prepare.bat` / `prepare.sh` (generates stubs and installs dependencies).

```bash
cp .env.sample .env   # adjust paths and ports
pnpm run build
pnpm run start
```

For the browser UI, start `browser-client` separately (see [`../readme.md`](../readme.md)).

Open `microservice_node_server.code-workspace` in VSCode and use the launch configurations to debug the server or regenerate ASN.1 stubs.

# Global managers

Singleton manager classes (prefix `the_`) are created in `src/globals.ts`:

```typescript
import {
	theClientConnectionManager, // connected WebSocket clients
	theConfig,                  // validated configuration from environment
	theLogger,                  // logging (console, file, …)
	theServer,                  // ASN.1 ROSE dispatch / handler routing
} from "./globals.js";
```

# Project structure

Starting point is `src/`:

```bash
|-- app.ts                 # entry: logging, singleton init, express startup
|-- express/
|   +-- expressInit.ts     # HTTP/HTTPS server, mounts routes
|   +-- expressRouter.ts   # auto-loads route modules
|   +-- routes/
|       +-- echoSample.ts       # GET/POST /echo (health probe)
|       +-- restSample.ts       # POST /rest/* stateless invokes
|       +-- webSocketSample.ts  # WS /ws stateful sessions + events
|-- handlers/
|   +-- settingsManager.ts    # asnGetSettings / asnSetSettings + asnSettingsChanged events
|   +-- eventManager.ts         # asnCreateFancyEvents + asnFancyEvent events
|-- lib/
|   +-- clientConnection.ts     # per-WebSocket client state, send/receive
|   +-- …
|-- singletons/
|   +-- config.ts, server.ts, clientConnectionManager.ts, …
|-- stub/                  # generated by esnacc (-RTS_SERVER); do not edit
```

Stub sources live in `samples/interface/*.asn1`. Run `prepare.bat`, `prepare.sh`, or `node prepare.js` from the `samples` folder.

# Examples

## REST (`/rest`)

Stateless HTTP POST. Suitable for invoke/result round-trips. Server-to-client events are **not** delivered on this path (see `eventManager` — `asnCreateFancyEvents` rejects REST-only calls).

## WebSocket (`/ws`)

Stateful connection. Each client gets a `ClientConnection` with a session ID in `theClientConnectionManager`. Events (`asnSettingsChanged`, `asnFancyEvent`) are dispatched only to connected WebSocket clients.

Useful breakpoints: `restSample.restRequest`, `clientConnection.wsClientMessage`, `settingsManager.onInvoke_asnSetSettings`, `eventManager.onInvoke_asnCreateFancyEvents`.

# SSL key and certificate

Optional TLS: set `MICROSERVICE_SERVER_LISTEN_PORT_TLS`, `MICROSERVICE_SERVER_CERTFILE`, and `MICROSERVICE_SERVER_KEYFILE` in `.env`.

```bash
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes
```

# Default configuration

Settings are read from the environment via `ucconfig` / `dotenv`. Copy `.env.sample` to `.env` in this directory.

```
MICROSERVICE_ENVIRONMENT=development
MICROSERVICE_LOG_LEVEL=debug
MICROSERVICE_LOG_TO_CONSOLE=1
MICROSERVICE_LOG_DIRECTORY=X:\tmp\microservice
MICROSERVICE_SERVER_FQDN=localhost
MICROSERVICE_SERVER_LISTEN_PORT_TCP=3020
# MICROSERVICE_SERVER_LISTEN_PORT_TLS=3021
# MICROSERVICE_SERVER_CERTFILE=cert.pem
# MICROSERVICE_SERVER_KEYFILE=key.pem
# Encoding: 1 = JSON, 2 = BER
MICROSERVICE_SERVER_DEFAULTENCODING=2
```

`MICROSERVICE_ENVIRONMENT` must be one of `development`, `staging`, or `production` (required by `ucconfig`).
